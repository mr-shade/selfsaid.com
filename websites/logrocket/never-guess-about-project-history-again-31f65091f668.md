_Never guess about project history again_

![](https://storage.googleapis.com/blog-images-backup/1*-0hawDLsHJI19yFsuGHU9Q.jpeg)

When it comes to productivity in software development, knowledge is power. The industry of project management tools‚Ää‚Äî‚Äähelping a development team to know where they stand‚Ää‚Äî‚Ääis vast. But software developers have one underused tool that can be leveraged to provide much more information‚Ää‚Äî‚Ää**version control.**

A life of a software project is punctuated by _releases_. In open-source, they can be publishings to a package manager. In a software shop, typically we‚Äôre talking about deployments to a production server.

Whatever they are, the knowledge that is of great use when it comes to releases is _when_ they have happened and _what_ they included. In a big enough project it‚Äôs common to keep a change log and tag releases, but who has time for that with when you have to [release early and release often](https://en.wikipedia.org/wiki/Release_early,_release_often)?

Usually, it looks something like this:

The last release happened a week ago, and now that we‚Äôre done with a new, necessary feature, let‚Äôs do another one.

The question of what else changed and the size/impact of how big the overall changes remains unanswered. In essence, we simply hope that we did not forget about anything important.

[![](https://storage.googleapis.com/blog-images-backup/1*GPjaPKNNUYHU8EsA3Z0JGA.png)](https://logrocket.com/for/bug-tracking-tools/)

### So let‚Äôs just automate all¬†that

It all starts with a **commit message**. Structured commit messages will be the very basis of the automation process. Another git entity as use will be a **tag**‚Ää‚Äî‚Ääwhich will mark each release. Based on the information in commit messages since last release, the next one will be prepared‚Ää‚Äî‚Ääor not, if the changes since last release won‚Äôt call for it. Neat, right?

_All that follows is available in this repository:_

> #### [GitHub ‚Äì adekbadek/semantic-release-demo: a minimal example of using semantic-release for release/version control](https://github.com/adekbadek/semantic-release-demo)
> 
> a minimal example of using semantic-release for release/version control ‚Äì adekbadek/semantic-release-demo

#### [Step 1](https://github.com/adekbadek/semantic-release-demo/commit/7de4583e2f3288fcc55ec10190d40525988a660e): Structured commit¬†messages

First off, a helping hand in structuring the commit messages: [commitizen](https://commitizen.github.io/cz-cli/). Install it with `$ npm i -D commitizen` and then set it up to use the [conventional changelog](https://github.com/conventional-changelog/conventional-changelog): `$ npx commitizen init cz-conventional-changelog -D -E`¬†. Then add an npm script that will call `git-cz` (the binary for commitizen) and after staging the changes in git, run it. Now you should see this welcoming prompt:

![](https://storage.googleapis.com/blog-images-backup/1*i-C_2JpIMgaRbVeaL3qa8g.png)

#### Step 2: Ensuring compliance with git¬†hooks

Good. But working with other people and your future self means that someone might forget to use commitizen and wreck the whole setup. Fortunately, git provides a thing called [hooks](https://git-scm.com/book/en/v2/Customizing-Git-Git-Hooks)! If you look into¬†`.git/hooks` of a fresh repository, you‚Äôll see a bunch of `*.sample` files‚Ää‚Äî‚Äähooks are simply bash scripts that will fire when certain actions occur. But who codes in bash when you have npm, right? Just `$ npm i -D husky` and watch that hooks folder populate with scripts that you don‚Äôt have to write. [Husky](https://github.com/typicode/husky) is a tool that does the hooks business based on config in package.json.

Now let‚Äôs get a linter for those commit messages. Firstly, install it‚Ää‚Äî‚Ää`$ npm i -D [@commitlint](https://twitter.com/commitlint "Twitter profile for @commitlint")/{config-conventional,cli}`. Then, create the `commitlint.config.js` config file:

module.exports = {extends: \['[@commitlint/config-conventional](https://twitter.com/commitlint/config-conventional "Twitter profile for @commitlint/config-conventional")'\]}

After that, provide some information for husky in package.json:

‚Äúhusky‚Äù: {
  ‚Äúhooks‚Äù: {
    ‚Äúcommit-msg‚Äù: ‚Äúcommitlint -E HUSKY\_GIT\_PARAMS‚Äù
  } 
}

That‚Äôs it! Try to add the files to git staging and commit with a gibberish commit message‚Ää‚Äî‚Äähusky‚Äôs gonna bite!

#### Step 3: Continuous Integration

Now we have a guarantee that the commit messages will be meaningful (or at least formatted). Time for the main protagonist of this tutorial: enter semantic-release: `$ npm i -D semantic-release`¬†. Along with that, add an npm script that will invoke it (just `‚Äúsemantic-release‚Äù: ‚Äúsemantic-release‚Äù`).

This tool is by default meant to publish on npm‚Ää‚Äî‚Ääif that‚Äôs not in your plans then some default-config-overriding will have to be performed. Add a `‚Äúrelease‚Äù` key in package.json with the following content:

"plugins": \[
  "[@semantic](https://twitter.com/semantic "Twitter profile for @semantic")\-release/commit-analyzer",
  "[@semantic](https://twitter.com/semantic "Twitter profile for @semantic")\-release/release-notes-generator",
  \[
    "[@semantic](https://twitter.com/semantic "Twitter profile for @semantic")\-release/npm",
    {
      "npmPublish": false
    }
  \],
  "[@semantic](https://twitter.com/semantic "Twitter profile for @semantic")\-release/github"
\]

Semantic-release will do its work in GitHub (and npm), operating from a CI server‚Ää‚Äî‚ÄäCircleCI in this case. It will thus need access to GitHub and npm via _tokens_, which have to be added to CI‚Äôs settings:

![](https://storage.googleapis.com/blog-images-backup/1*UU1v3xMShYEaK0jq2zix3w.png)

(ignore the NPM\_TOKEN if you don‚Äôt plan to publish on¬†npm)

As you can see, the tokens need be available as environment variables named `GITHUB_TOKEN` and `NPM_TOKEN`. Now let‚Äôs add a config for CircleCI, called¬†`.circleci/config.yml`‚Äî which is a bit bloated so you may just [copy it from here](https://github.com/adekbadek/semantic-release-demo/blob/master/.circleci/config.yml).

Everything is ready now! If CI sees a commit message that should trigger a release (like those starting with `feat` or `fix`), all will happen automatically. A release and tag will be published on GitHub and‚Ää‚Äî‚Ääunless configured differently‚Ää‚Äî‚Ääa new package version on npm.

#### Step 4: Generating a change¬†log

But, there is one problem. Two, actually. After a release, the `version` field in `package.json` has remained the same, and where is that change log that will be so helpful down the road? Don‚Äôt worry, it‚Äôs all two changes away:

First, install some additional packages: `$ npm i -D [@semantic](https://twitter.com/semantic "Twitter profile for @semantic")-release/changelog [@semantic](https://twitter.com/semantic "Twitter profile for @semantic")-release/git`. Then, add this config in `‚Äúrelease‚Äù` section of `package.json`:

"prepare": \[
   "[@semantic](https://twitter.com/semantic "Twitter profile for @semantic")\-release/changelog",
   "[@semantic](https://twitter.com/semantic "Twitter profile for @semantic")\-release/npm",
   {
    "path": "[@semantic](https://twitter.com/semantic "Twitter profile for @semantic")\-release/git",
     "assets": \[
      "package.json",
      "package-lock.json",
      "CHANGELOG.md"
    \],
    "message": "chore(release): ${nextRelease.version} \[skip ci\]nn${nextRelease.notes}"
  }
\]

This will tell `semantic-release` to generate a change log and then to add a commit after it‚Äôs done with the version releasing. You can tweak many more settings in this section, visit [semantic-release docs](https://semantic-release.gitbook.io/semantic-release/) for more info.

With that setup, a release will happen not on a whim, but based on the changes in code. And when it does, a change log will be generated so everyone knows _what_ landed in the subsequent releases and _when_ they happened.

### What do these version numbers¬†mean?

As you probably noticed, a release is marked by _a version_. With semantic-release, the numbers in the version follow a system called [Semantic Versioning](https://semver.org/) (aka SemVer). In short, these denote **major**, **minor**, and **patch** versions. A patch number is incremented when backwards-compatible bugfixes are added, minor: backwards-compatible features, and major: breaking changes (incompatible with the versions before).

Semantic versioning is important for your users (or a client)‚Ää‚Äî‚Ääthis way they know what can they expect with a new release and what has changed since they last used/seen the project. If the last version someone used is `1.0.1` and the current version is `1.1.42` then they know that there were many more bug fixes than features added in the meantime.

Semantic versioning can be opposed to [sentimental versioning](http://sentimentalversioning.org/), in which the version numbers are incremented in an incoherent manner, and in effect they don‚Äôt mean anything.

### Why not?

All that setup might seem like an overkill, and for a small project maintained by a single person it might be. But I have thanked myself many times for setting it up at the beginning and I‚Äôm sure I‚Äôll use it again whenever I can. It‚Äôs just very reassuring to have that much information in git, where it won‚Äôt become outdated and is strictly tied to the code.

As for other qualms:

> I just need a glance at the git log diff and I‚Äôm good.

Well, a log is by nature not meant to be read in entirety. It has a very weak signal to noise ratio.

> Structuring commit messages is just annoying!

Having an unreadable commit history is even more. Try debugging with no information whatsoever about when a bug could have been introduced.

> I‚Äôm pretty used to meaningless commit messages‚Ä¶

Having to specify what the commit is about makes a developer think twice about the committed changes. If you‚Äôre unsure about the type of a commit, maybe the changes aren‚Äôt so clear either?
