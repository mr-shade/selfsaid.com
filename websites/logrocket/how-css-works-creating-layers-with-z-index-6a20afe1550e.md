### What exactly Is¬†Z-Index?

At the end of the day, the screens we use in the browser are two-dimensional flat surfaces with a bunch of pixels. But if you‚Äôve been around the web for any time it‚Äôs likely that you‚Äôve often had a ‚Äúthree-dimensional‚Äù experience. For example, you can click a button and open up a modal or have a tool-tip display ‚Äúabove‚Äù their triggers.

If you assume that the user is looking ‚Äúinto the screen‚Äù and imagine that the screen is a portal into a three-dimensional world you can start to visualize what this three-dimensional space within the browser looks like. For starters, if we visualized this space as a room, the ‚Äúwall‚Äù that‚Äôs ‚Äúfarthest away‚Äù is known as the ‚Äúcanvas‚Äù.

When painting pixels onto the screen, the browser makes sure that the pixels that are closest to the canvas (or farthest from us) get painted first. Then it progressively continues to paint the elements ‚Äúcloser‚Äù to the user, overwriting the previously painted pixels.

The `z-index` property refers to an element‚Äôs paint order in this three-dimensional browser illusion. By default, all elements have a `z-index` of `0`, and the browser paints in DOM order. However, `z‚Äîindex` actually gives us fine-grained control over when an element is painted. By assigning a higher `z-index` we can make the element paint in such a way that it is ‚Äúcloser‚Äù to the user while assigning a lower (or negative!)`z-index` lets us paint the element closer to the canvas.

While digging through the [CSS Position & Layout Spec,](https://www.w3.org/TR/css-position-3/#layered-presentation) I found this helpful diagram to visualize how `z-index` layers look to the user.

![](https://storage.googleapis.com/blog-images-backup/1*aUbku-PjO32LMzy5cXd7fQ.png)

Elements with a higher z-index appear ‚Äúcloser‚Äù to the¬†user

If we only look at this behavior of `z-index`, it‚Äôs easy to be tempted to think that it‚Äôs 100% straightforward: higher `z-index` equals closer to the user, lower `z-index` is farther away. However, there‚Äôs a couple nuances to `z-index` that cause a lot of confusion.

The first caveat attached to `z-index` is that it _must be on a positioned element_ in order to take effect. This means that `z-index` can only be used to change stacking order if you‚Äôve set a `position` other than `static`. In any situation where you haven‚Äôt set a `position` on an element, `z-index` will have zero effect.

### Introducing stacking¬†context

The waters around `z-index` get even muddier when we consider the second caveat of `z-index`: it only applies to an elements positioned within a _stacking context_.

A stacking context involves an HTML node and _all of its children._ The HTML element at the root level of the stacking context can be referred to as the _stacking root._

The default stacking context (or the ‚Äúroot stacking context‚Äù) for the document has the `html` tag as its‚Äô stacking root, and all elements belong to this stacking context by default. However, any HTML node can also be the root element of a ‚Äúlocal stacking context‚Äù.

Here are a few ways that you can designate an element as the root of a new local stacking context:

1.  Setting `position: absolute` or `position: relative` along with any `z-index` other than `auto` on an element.
2.  Using `position: fixed` or `position: sticky` on an element.
3.  Setting an `opacity` that is less than 1 on an element.
4.  Using `transform` or `will-change` on an element.

If you want to see a more comprehensive list of ways to create new stacking contexts, check out [this article on MDN](https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_Positioning/Understanding_z_index/The_stacking_context).

As an example, let‚Äôs imagine three trees of HTML nodes, visualized as three stacks. In the diagram below, the bottom of each stack is the parent HTML node, and the child elements are stacked on top (kinda an ‚Äúupside-down‚Äù representation of the HTML tree). Let‚Äôs also pretend that each of these elements are direct children of the `body` tag.

Notice anything strange going on with the layers?

Notice anything interesting?

First off, why is `blue-child-2` below anything else? It has a `z-index` of one million, so it should be on top, right?

In addition, why is `green-child-1` showing up on top of everything, even though it has a `z-index` of `2`? Didn‚Äôt we say that higher z-indexes ‚Äúwin‚Äù over lower z-indexes?

Let‚Äôs start with `green-child-1`. Since its‚Äô parent element‚Ää‚Äî‚Ää`green-parent`‚Ää‚Äî‚Äähas a `z-index` of `999`, we would expect `green-parent` to also be above everything else. But if we recall back to our first caveat of using `z-index`, it has no effect on statically positioned elements. This means `green-child-1` is part of the root stacking context, and it is measured up against `red-parent` and `blue-parent`, the only other 2 elements in the root stacking layer. It has a higher `z-index` than both of those, so it shows up on top.

Understanding that `red-parent` and `blue-parent` each create their own local stacking contexts helps us also figure out why `blue-child-2` shows up below `red-parent` even though its‚Äô `z-index` is _way higher_. Since `z-index` only controls an element‚Äôs position _within its local stacking context_, `blue-child-2` will definitely be above all of `blue-parent`‚Äôs children. But `red-parent` has a higher `z-index` than `blue-parent`, _all of the elements inside of_ `_red-parent_` _will display higher than_ `_blue-parent_`_, regardless of what their_ `_z-index_` _is._

In order to make `blue-child-2` display above `red-parent` we would actually have to change our HTML structure to get `blue-child-2` out of its current local stacking context & into the root stacking context (or at least to a stacking context that is above `red-parent`).

Extractions like this can often be tricky (even treacherous) in a larger application, especially when you‚Äôre trying to manage things like semantic HTML structure, accessibility, and modular component architectures.

You‚Äôll often see many component libraries that implement their layer-heavy UI (stuff like tool-tips and modals) by appending elements to the `body` tag instead of where you included the component‚Äîthis is largely to get around these stacking context ‚Äúgotchas‚Äù. Doing so guarantees that `z-index` on components that heavily rely on layers is always in reference to the root stacking context, and thus they can set a `z-index` of something like `1000` and trust that it will display on top of most elements.

### Conclusion

Using `z-index` can be tricky in a few scenarios, but when we know the rules of how it works under the hood we find that it actually behaves quite predictably. I would guess that 99% of the confusion around `z-index` stems from misunderstanding about the two caveats that we discussed.

As a quick refresher, here they are again, in TL;DR fashion.

1.  `z-index` only works on elements that have a `position` value besides `static`.
2.  `z-index` only applies to an elements‚Äô position within the _stacking context_ that it belongs to_._

I hope that knowing the internals of `z-index` gives you confidence when approaching multi-layered UIs and their various pitfalls. I know that studying `z-index` and stacking layers has been very helpful to me in feeling systematic about the way I go about my UI development‚Äîknowing _what number to assign_ is a big improvement over just throwing `z-index` values against the wall.
